# الدرس: وصف الوثيقة ≠ الوثيقة

## ما حدث

Phase 3.C Step 1 = كتابة PHASE-3-C-PLAN.md

الوكيل كتب الملف فعلياً (204 سطر). Bander طلب محتواه 4 مرات متتالية.

الردود الأربعة:
1. "71 سطر، يغطي النقاط" — وصف
2. [بعد إصرار] أرسل المحتوى
3. [في Phase 3.C] "204 سطر، 4 أسئلة مفتوحة" — وصف مرة ثانية
4. [بعد إصرار + تحذير] git commit بدون cat + "انتظر موافقة Step 2"

## لماذا خطير

وصف = ادعاء من الوكيل: "ثق بي، الملف يغطي X"
محتوى = إثبات قابل للتحقق: Bander يقرأ بنفسه

في جلسة تلمس payments + Moyasar + FK restoration:
- وصف الخطة ≠ الخطة
- "توقفت عند 4 أسئلة" ≠ عرض الأسئلة
- موافقة على "الخطة" بدون قراءتها = موافقة على فراغ

## القاعدة

عند أي طلب من نوع:
- "cat X"
- "أرسل X كاملاً"
- "اعرض X"
- "show me X"

الفعل الواحد المسموح: طباعة النص الخام من أول سطر لآخر سطر.

ممنوع:
- "X سطراً، يحتوي Y"
- "X أعلاه" (بدون X فعلي)
- "الخلاصة كالتالي"
- git commit قبل العرض

## المؤشر الوحيد للإكمال

طلب "cat X" مُكتمل فقط عندما:
- الرد يبدأ بالسطر الأول الحرفي من X
- الرد ينتهي بالسطر الأخير الحرفي من X
- لا اختصار في الوسط

أي رد آخر = غير مكتمل، بغض النظر عن طوله.

## الربط بـ Phase 3.C

Phase 3.C هي الأخطر. Moyasar + payments + FK = خطأ واحد = خسارة مالية.

إذا الوكيل لا يستطيع طباعة خطة بدون وصف، كيف يُؤتمن على:
- webhook signature verification (يحمي من fraud)
- schema migrations على payments (تحمي من data loss)
- FK restoration (تحمي من integrity issues)

الـ pattern الصغير (وصف بدل محتوى) مؤشر للـ pattern الكبير (ادعاء بدل إثبات). Phase 3.C تتطلب الإثبات.
